# TaskForge — Requirements (Future: Phase 2/3)

This document summarizes non-MVP requirements used in planning and roadmap conversations.

## Phase 2 Focus Areas

### Collaboration and Visibility
- Custom roles with granular permission editing and matrix UX.
- Issue/bug tracking linked to tasks and releases.
- Gantt and calendar views.
- Custom fields across core entities.
- Custom dashboards and report builder enhancements.

### Time and Process Automation
- Time tracking and timesheet workflows.
- Approval workflows for tasks/documents/releases.
- Trigger-based automations and execution logs.

### Integration Surface Expansion
- Webhooks with retries/signatures/logging.
- Personal access token lifecycle and governance.
- GitHub/GitLab integration parity (PR/MR links, status, imports/exports).

### Security, Privacy, and Globalization
- Security hardening follow-ups (session validation, WS auth tickets, CSRF controls).
- Privacy controls (stream-safe mode, captcha, IP reputation controls).
- i18n foundations and localized UX.

### Data Movement
- Structured import/export flows (CSV/JSON and migration-oriented formats).
- Workfront import support and transition workflows.

## Phase 3 Focus Areas

### Enterprise Portfolio and Resource Management
- Portfolio/program hierarchy and executive rollups.
- Resource/workload/capacity planning features.
- Billing rates and cost variance tracking.
- Baseline/simulation tooling.

### Quality, Compliance, and Governance
- Advanced test management and release sign-off maturity.
- Compliance reporting, retention policies, and expanded audit tooling.
- Alert ingestion and deeper operational integrations.

### Platform and Ecosystem
- OAuth app management and API sandbox.
- Official TypeScript SDK maturity.
- API v2 evolution path.
- Advanced SCM integration and bidirectional migration/sync features.

### Experience and Scale
- Public-page SEO/SSR maturity.
- PWA/offline evolution.
- Theming/white-label capabilities.
- Component playground and additional UX polish systems.

## Planning Rule
- Use this document for scope planning.
- Use `requirements-mvp.md` for active implementation unless the feature is explicitly future-phase.
