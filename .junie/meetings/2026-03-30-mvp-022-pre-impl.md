## 1. Router: Code-Based vs. File-Based

**Priya:** TanStack Router v1 supports both file-based routing (via `@tanstack/router-plugin/vite`) and code-based routing. File-based generates `routeTree.gen.ts` at dev/build time â€” it doesn't exist until the first `pnpm dev` run, so tests would fail during CI. Code-based routing is fully explicit, doesn't require the build plugin, and is 100% testable without running Vite.

**Priya:** I recommend code-based routing for this implementation. The route tree lives in `src/router.tsx` and is imported directly in `App.tsx`. We can migrate to file-based later as a pure refactor â€” routes won't change.

**Decision:** Code-based TanStack Router. Route tree in `src/router.tsx`. `@tanstack/router-plugin` stays in `package.json` for future migration but is NOT added to `vite.config.ts` today.

## 2. Protected Route Pattern

**Lena:** On a 401/unauthenticated redirect, the user should be sent back to where they were. The `redirect` search param should carry the original path.

**Priya:** TanStack Router's `beforeLoad` runs before mount. I'll throw a `redirect({ to: '/auth/login', search: { redirect: location.pathname } })`. On successful login, the store navigates to `search.redirect ?? '/dashboard'`.

**Finn:** Don't flash the protected page before the redirect fires. `beforeLoad` is synchronous for the auth check (reading Zustand state), so there's no flash.

**Decision:** `beforeLoad` in `_authenticated` route checks `useAuthStore.getState().isAuthenticated`. Auth pages redirect away if already authenticated (prevents back-button after logout). Login posts to `/auth/redirect` param on success.

## 10. Token Refresh Flow

**Priya:** The refresh token is in an HttpOnly cookie (set by the backend). The API client calls `POST /api/v1/auth/refresh` with `credentials: 'include'` on 401. The response returns a new `accessToken`. The client updates the Zustand store and retries the original request exactly once. If refresh fails (401/403), it calls `clearAuth()` and navigates to login.

**Priya:** To prevent refresh loops, I'll use an `isRefreshing` flag and a `pendingRequests` queue. While a refresh is in flight, subsequent 401s wait for the result and retry with the new token.

**Decision:** Full refresh flow with `isRefreshing` flag + queue. On refresh failure: clear auth + redirect to login.
