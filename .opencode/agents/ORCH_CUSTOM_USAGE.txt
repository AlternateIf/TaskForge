ORCH-CUSTOM: HOW TO USE

Purpose
- Copy-paste prompt templates for validating and running the orchestration workflows.
- Keep this file close to the agent definitions for quick iteration.

Recommended Run Order
1) 4A (routing smoke tests)
2) 4B (bounded bugfix flow)
3) 4C (feature flow with frontend prototypes)
4) 4D (backend-only feature flow)
5) 4E (validation gate only)

4B) Bounded Bugfix Flow

Prompt:
Bugfix workflow. Delegate-only mode.

Issue: ally leads to a rate limit (429)
Scope: apps/api auth + apps/web auth probably.

Mandatory rules:
- Delegate-only: do not use local read/grep/glob/list/symbol tools in orch-custom.
- First response must dispatch at least one task (or ask max 3 targeted clarifying questions).
- If endpoint/error/steps are concrete, first response must dispatch at least one implementer repro task; explorer can run in parallel but must not block repro start.
- Start concrete reproduction within 10 minutes.
- Before first repro attempt: max 12 file reads and max 15 search/grep calls (by subagents).
- Reproduction cap: max 15 minutes OR max 3 attempts.
- Do not run environment bootstrap (docker/startup/seed) unless health check fails; then ask first.
- If not reproducible, STOP and return NON_REPRO_REPORT with:
  1) exact attempts made (commands, inputs, timestamps),
  2) likely hypotheses,
  3) instrumentation/logging additions,
  4) smallest safe hardening patch option.
  Then ask for my approval before continuing.
- Do not treat explorer hypotheses as reproduction proof; reproduction must be confirmed by implementer/tester execution output.

After successful repro:
follow process exactly:
reproduce -> minimal fix -> test -> unit tests if needed -> review -> docs/contracts if backend behavior changed -> pnpm lint && pnpm test.

Parallelization:
- Dispatch independent tasks in the same turn.
- Do not serialize independent delegations.

Merge policy:
- Manual merge only.

Operator override (when needed):
strict attempt mode now: no explorer tasks, run remaining repro attempts only via backend-implementer/frontend-prototyper-implementer, then stop with NON_REPRO_REPORT.

4C) Feature Flow (Frontend + Backend + Prototypes)

Prompt:
Feature workflow with frontend + backend. Delegate-only mode.

Rough spec:
We need a page that is only available with the correct permissions (or roles that hold permissions). [a rough overview of permissions can be found in setup.md]
This page features a) organization settings like changing the org name, showing all members of the org, allowing to remove members
as well as inviting new members. the flow is: if registration is open [can be set via env variable] user can just register; if it is closed the "registration" is invite only.
For that we previously already had the frontend to invite members by email [with resend functionality etc /auth/invite?token=]. they would get an email with a link and through that they could register. maybe you can find the old behavior via git
b) role and permission handling where you can create new roles, assign roles to users of an org [keep in mind a user can have multiple orgs] and assign single permissions to users of an org
everything should feel intuitive and easy for users. because this is important i want you to create 6 prototypes this time. org settings may also include name, logo, auth settings (force mfa)
for the frontend consider mobile friendly, dark and light mode and give some variations in the prototypes. the prototypes should be html to be viewable in the browser. it would be nice if the prototype had a theme toggle

Mandatory process:
1) Plan with me first (planner). Create one shared `PLAN_MD_PATH` in `.ai/plans/` before round 1, then keep updating that same file each round.
   If planner asks clarification questions, ask me and feed answers back to planner before moving on.
2) Challenge the plan (plan-challenger).
   Always send challenger the full current plan from `PLAN_MD_PATH` plus a short "changes since last round" summary.
3) Send challenger findings back to planner only while challenger has unresolved HIGH issues; stop early when highest severity is below HIGH.
   Max 10 loops applies only to those blocking MEDIUM-severity rounds.
   If I explicitly ask, increase planning loop cap to 15 for the current run only.
4) Stop for my manual plan review feedback using the same `PLAN_MD_PATH`.
5) If I provide feedback, run planner + plan-challenger for up to 3 additional rounds to incorporate/challenge my input.
   Stop early when challenger has no HIGH issues, and keep updating the same `PLAN_MD_PATH`.
   Pass this same `PLAN_MD_PATH` to implementers, testers, and `unit-test-agent` in all subsequent delegated tasks.
6) Run frontend/backend discovery in parallel only if PLAN_READY still has unresolved unknowns.
7) Create exactly 3-4 frontend prototypes under (if necessary - only for frontend):
   local/prototype/%feature%/html-prototypes/
   Each prototype must include a light/dark switch and be mobile friendly (minimum 360px viewport support).
   Prototype artifacts must be browser-viewable `.html` files (not JSX/TSX).
   Prototype generation must not change existing code; only files in the prototype directory may be edited.
8) Run prototype compliance check and report exact prototype file paths.
9) Stop and ask me to choose MVP prototype.
10) Implement chosen MVP and backend support in parallel where independent.
    For every frontend task, pass `MVP_PROTOTYPE_HTML_PATH`.
    If prototype revisions were requested after selection, pass `REVISED_MVP_PROTOTYPE_HTML_PATH` and treat it as authoritative.
    Require `PROTOTYPE_ADHERENCE_REPORT` in frontend implementer output.
11) Check plan coverage against acceptance criteria.
12) Run frontend/backend reviews in parallel (`reviewer-frontend` + `reviewer-backend`), then merge findings with `reviewer-full` if needed.
13) Run frontend/backend targeted tests in parallel (`tester-frontend` + `tester-backend`), then run full gate once with `tester-full`:
   pnpm lint && pnpm test
14) Update docs via `docs-contract-agent`; if backend contract changed, update swagger/openapi artifacts there too.
15) Manual merge only.

Boundaries:
- No local tool work in orch-custom; delegate all execution.
- No environment bootstrap (docker/startup/seed) unless health check fails, then ask first.
- If blocked, return concise blocker + required decision.

  Permission Gating Audit (Batch=10, Delegate-Only)

  Objective:
  Audit permissions in batches of 10, identify frontend/backend places that should be gated, ask user for per-
  permission decisions, then implement and validate.

  Mode:
  - Delegate-only orchestration.
  - No local read/grep/glob/list/symbol work in orch-custom.

  Scope:
  - Process permissions in user-provided order, sequential batches of 10.
  - Discovery runs in parallel.
  - Implementation runs in parallel only for disjoint file sets; if overlap exists, enforce sequential execution for
  overlapping items.

  Batch workflow (strict gates):
  1) Select next 10 permissions.
  2) For each permission, run 2 parallel discovery tasks:
  - `explorer` Round A
  - `reviewer-frontend` + `reviewer-backend` (parallel) for “missing gate” candidate detection
  3) Aggregate into `PERMISSION_GATING_REPORT` per permission:
  - candidate place/feature to gate
  - exact code locations
  - current coverage: frontend/backend/both/missing
  - rationale for gating
  - confidence: high/medium/low
  - overlap/alias note (e.g. `*.manage.*` vs granular permission)
  4) Ask user for per-permission decision (all 10 at once):
  - `GATE`
  - `NO_GATE`
  5) Execute changes:
  - `GATE`: delegate to frontend/backend implementers to add gating and wiring
  - `NO_GATE`: no code changes
  - Parallelize only when file ownership is disjoint
  6) Validation loop (changed scopes only):
  - run `tester-frontend` / `tester-backend` in parallel where applicable
  - run lint + tests
  - if failing, route fixes back to implementers and repeat until green
  7) Unit tests:
  - delegate `unit-test-agent` to add/update tests for each newly added gate
  8) Documentation:
  - delegate `docs-contract-agent` to append permission gating notes to `.ai/permissions.md` (create if missing)
  9) Batch completion gate:
  - all 10 permissions = implemented (if needed), tests green, docs updated
  - then move to next batch

  Mandatory rules:
  - First response must dispatch tasks (no analysis-only response).
  - Dispatch independent tasks in the same turn.
  - No bootstrap (docker/startup/seed) unless health check fails; ask first.
  - Manual merge only.

  Required output format:
  `PERMISSION_GATING_REPORT`
  - Permission
  - Existing gating locations
  - Missing gating candidates
  - FE/BE coverage
  - Rationale
  - Confidence
  - User decision
  - Implementation status
  - Test status
  - Docs status
  - Final status: COMPLETE | BLOCKED
